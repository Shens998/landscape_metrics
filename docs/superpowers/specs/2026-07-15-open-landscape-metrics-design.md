# Open Landscape Metrics｜设计说明

> 状态：用户已批准，待书面审阅。
>
> 日期：2026-07-15
> 范围：公开、开放科学、独立实现的 Python 景观格局指标库。

## 1. 目标与边界

项目将开发一个面向土地利用/土地覆被（LULC）**分类栅格**的开源 Python 库。它提供可从 Python 脚本直接调用的 API，计算常用的斑块级、类别级和景观级格局指标。

本项目采用公开发表的指标定义作为规范来源，独立撰写代码、测试、文档和示例。项目不读取 FRAGSTATS 工程文件、不调用其可执行程序、不复制其界面或文档文字，也不承诺与其产生逐字或逐值兼容。文档可在学术脉络中引用 FRAGSTATS 与相关文献，但应清楚说明这是独立实现。

“纯 Python”意指计算引擎不依赖专有外部软件；允许使用成熟 Python 科学计算与 GeoTIFF 读写依赖（例如 NumPy、Rasterio/GDAL）。

v0.1 不包括：命令行工具、图形界面、矢量面数据输入、移动窗口分析、FRAGSTATS 工程兼容、全局距离场指标（如 Core Area）或图论连通性指标（如 IIC/PC）。

## 2. 输入、数据规则与 API

### 2.1 输入

v0.1 接受：

- 单波段分类 GeoTIFF；
- 作为高级入口的二维整数 NumPy 数组，须同时提供像元尺度或等价空间元数据。

浮点连续表面、混合类型波段和未说明类别含义的数组会被拒绝。背景类别是一个有效的显式类别；NoData 是不参加统计的缺失区域，两者不得混同。库不得隐式重采样或重投影。

长度、面积和边缘指标要求投影坐标系，或由调用方明确传入等效的像元宽高。对以经纬度（度）为单位且未给出等效尺度的 GeoTIFF，库必须报错并说明原因。v0.1 仅接受北向上、无旋转或剪切的仿射变换；像元可以是长方形，但面积和共享边长必须分别依据像元宽高计算。

### 2.2 对外 API

首版以小而稳定的 API 为目标：

```python
from landscape_metrics import Landscape

landscape = Landscape.from_geotiff("lulc.tif", connectivity=8)
landscape_metrics = landscape.metrics()
class_metrics = landscape.class_metrics()
patch_metrics = landscape.patch_metrics()
```

结果为 pandas DataFrame，并附带可序列化的计算元数据，包括输入摘要、CRS、像元尺度、NoData 规则、背景类别、连通规则、边界规则、执行路径、块大小、包版本和指标版本。

## 3. 模块架构

```text
GeoTIFF / NumPy 分类数组
          ↓
io：读取、元数据与输入验证
          ↓
topology：斑块、边缘、类别邻接、基础计数
          ↓
metrics：斑块级 → 类别级 → 景观级纯函数
          ↓
api：Landscape 对象与结构化结果
          ↓
validation：金标准、性质测试、分块一致性与性能基准
```

- `io` 只负责读取和验证，绝不计算指标。
- `topology` 负责可复用的斑块标记、边缘和邻接基础量。
- `metrics` 中的每项指标为单独、可单测的函数；不应隐藏 I/O 或全局状态。
- `api` 只编排输入、执行策略和结果，不重写指标公式。
- `validation` 保存公开的微型栅格、预期输出和性质测试。

这种边界允许替换内部算法而不改变用户 API，也使每项指标和每种数据规则可以独立验证。

## 4. 大型 GeoTIFF：结果严格一致的分块计算

库提供两种执行路径：

1. **内存路径**：将较小栅格读入内存并直接计算。
2. **分块路径**：按 Rasterio window 逐块读取大型 GeoTIFF，使用磁盘临时目录保存必要中间信息。

分块路径采用精确两阶段策略：

1. 每块完成局部斑块标记、类别面积和单向的共享边邻接（只统计右/下边，避免重复）；记录相邻块边界上的局部斑块对应关系。
2. 以并查集（union--find）合并跨块连通斑块，再按全局斑块身份汇总面积、周长、形状和类别邻接。

斑块连通与边共享邻接是两个明示的概念：`connectivity=4` 只跨块合并上下左右相接像元；`connectivity=8` 还必须比较边界两侧的三个相对位置，并处理四块交汇处的对角连接。边缘与类别邻接指标只计共享边，不把对角接触当作边。这些规则进入指标元数据和测试矩阵。

该算法的设计要求是：对 v0.1 支持的每个指标，整图与任意合法分块方案的结果必须严格相同。临时空间不足、不能完成跨块合并、输入在读取时变化等情况必须失败并保留诊断信息，不能回退为静默近似。

块大小可由用户设置；自动策略由内存预算推导，并在元数据中记录。v0.1 不实现需要全局距离变换或图论求解的指标，避免以局部缓冲产生未经声明的近似。

## 5. v0.1 指标范围

首版限定在大约 20 个基础指标；具体名称、公式和测试值须在实现计划中冻结。

| 层级 | 拟支持指标族 |
|---|---|
| 斑块 | 面积、周长、形状指数、周长面积比、分形维数、回转半径 |
| 类别 | 类别面积、景观占比、斑块数、斑块密度、最大斑块指数、总边缘、边缘密度、面积均值、面积离散度、形状均值、聚合度 |
| 景观 | 总面积、斑块数/密度、边缘密度、最大斑块指数、香农多样性、香农均匀度、辛普森多样性、类别聚合度摘要 |

指标在冻结前必须逐项确认：分母、单位、零值情形、单类别情形、NoData 和景观外边界的处理。若某名称存在多种通行公式，v0.1 只能选定一种，并在指标卡中明示。

## 6. 指标卡、文献与可引用性

每项指标都提供一个机器可读指标卡，并在文档中渲染。指标卡至少包含：

- 指标标识、名称、计算层级、公式、单位和有效范围；
- 所需基础量、连通规则、边界规则和 NoData 规则；
- 完整参考文献、DOI 或稳定链接；
- 原始来源标记：`original`、`definition_reference` 或 `review`；
- 实现函数、单元测试和引入该定义的版本。

不将权威软件手册误称为原始文献；无法可靠确认原始论文时，以定义性或综述性来源诚实标示。

## 7. 错误处理和可重复性

库使用明确异常类型，至少区分：输入分类错误、空间元数据不足、NoData/背景冲突、无效连通规则、分块临时空间不足和不支持的指标。错误消息需要包含可执行修复建议。

所有结果都可导出其完整配置与输入摘要。默认严格模式拒绝模糊输入，不做静默重编码、重投影、近似或单位猜测。

## 8. 验证、测试与发布门槛

每一个指标必须通过：

1. 人工可验算微型栅格的金标准测试；
2. 性质测试，例如类别重命名不改变景观级结果、旋转后面积不变等；
3. 内存路径与多种块大小的分块路径严格一致性测试；
4. 与至少一个独立公开参考实现或公开公式计算的交叉核对（仅作为外部诊断，不是兼容性承诺）；
5. 大型合成栅格的性能与峰值内存基准。

只有当指标卡、公式审查、测试和分块一致性均完成时，指标才可进入稳定 API。任何可能改变数值的规则修改必须提升相应版本、增加回归用例，并在变更日志中说明。

## 9. 开源发布与版本策略

建议采用 MIT 或 BSD-3-Clause 许可证；最终许可证、文档许可证、测试数据许可证和第三方依赖声明必须在发布前分别明确。发布仓库应包括：

- 安装与快速开始；
- API 参考与指标卡；
- 方法学边界、引用方式与贡献规范；
- 公共测试栅格和预期结果；
- 变更日志、版本策略和可复现实验环境。

v0.1 的稳定承诺仅限于上述分类 GeoTIFF、Python API、已冻结的基础指标及内存/分块一致性。移动窗口、CLI、矢量输入、核心面积和图论连通性指标将另行设计，不在 v0.1 范围内。

## 10. 设计自审结果

- **范围**：已通过只支持分类栅格、Python API 和基础指标限制为可分阶段实现的首版。
- **算法一致性**：大文件策略明确要求并查集合并和整图/分块严格一致；需要全局距离或图论的指标已排除。
- **科学可追溯性**：指标卡必须区分原始文献与定义性来源，避免虚假归因。
- **许可边界**：项目为独立实现，不使用专有引擎或其专有工程格式；开源发布前仍需确认自有代码、文档、测试数据和依赖的许可证。
- **未决但不阻塞设计的事项**：包名、最终开源许可证、具体约 20 个指标的冻结清单和外部参考实现的选择，将在实施计划阶段锁定。
